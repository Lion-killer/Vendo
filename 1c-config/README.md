# Міні-конфігурація 1С — бекенд Vendo

Окрема **конфігурація-постачальник** для **1С «Управління торгівлею для України»**,
яка виступає реальним бекендом мобільного застосунку Vendo замість демонстраційного
Node-сервера (`../backend/`).

> **Зміна архітектури.** Раніше це було розширення (CFE). Цільова платформа
> **не підтримує розширення**, тому ті самі об'єкти оформлено як окрему
> міні-конфігурацію, яку **вбудовують у цільову УТ через «Порівняння/об'єднання
> конфігурацій»**.

- Префікс об'єктів: `венд_`
- Сумісність: `Version8_3_19`
- Цільова база: УТ для України 2.3

## Склад

```
src/
├─ Configuration.xml                       # маніфест конфігурації-постачальника
├─ Catalogs/венд_МобильныеУстройства.xml   # довідник «Мобільні пристрої»
├─ HTTPServices/венд_МобильноеПриложение/  # HTTP-сервіс (REST API)
│  └─ Ext/Module.bsl                        # обробники методів
├─ Roles/венд_ОсновнаяРоль/Ext/Rights.xml  # права на власні об'єкти
└─ Languages/Русский.xml
```

## ⚠️ Важливо: не валідовується/не збирається standalone

Об'єкти посилаються на типи цільової бази (`CatalogRef.Склады`, `.Контрагенты`,
`.Номенклатура`, `.ТипыЦенНоменклатуры`, `.Пользователи`, `Документ.ЗаказПокупателя`,
регістри `ТоварыНаСкладах` / `ТоварыВРезервеНаСкладах` / `ЦеныНоменклатуры` /
`ВзаиморасчетыСКонтрагентами`). Цих типів немає в порожній конфігурації, тому
**завантажувати й перевіряти ці XML можна лише всередині копії цільової УТ**.
`cf-validate` тут перевіряє лише структуру власних об'єктів.

## Як вбудувати в цільову УТ

1. Зробити копію бойової конфігурації УТ (тестова база).
2. Завантажити в неї XML-джерела цих об'єктів (Конфігуратор / `LoadConfigFromFiles`
   у тимчасову конфігурацію-постачальника, далі **Порівняння й об'єднання
   конфігурацій** → перенести `венд_*` об'єкти).
3. Оновити конфігурацію БД, опублікувати HTTP-сервіс на веб-сервері (Apache/IIS) —
   отримати URL виду `https://<сервер>/<база>/hs/vendo`.
4. Завести записи довідника «Мобільні пристрої»: склад / тип цін / менеджер,
   контрагенти й товари, `ИдентификаторУстройства` (GUID для QR).

### Передумова сумісності
HTTP-сервіс вимагає **платформи ≥ 8.3.5**. Режим сумісності самої конфігурації
може лишатися нижчим — перевірено: HTTP-сервіс додається в конфігурацію в режимі
сумісності **8.2.13** на платформі BAF 8.3.19.1529. Тобто об'єднання з УТ (8.2.13)
обмежень тут не має.

## Довідник «Мобільні пристрої» (`венд_МобильныеУстройства`)

Центральна сутність: один запис = один планшет/користувач застосунку.

| Реквізит | Тип | Призначення |
|---|---|---|
| `ИдентификаторУстройства` | Строка(36), індекс | GUID із QR-коду для автентифікації |
| `Склад` | СправочникСсылка (будь-який) | джерело залишків (обирати елемент `Склады`) |
| `ТипЦен` | СправочникСсылка (будь-який) | тип цін (обирати `ТипыЦенНоменклатуры`) |
| `Менеджер` | СправочникСсылка (будь-який) | відповідальний (обирати `Пользователи`) |
| ТЧ `Контрагенты.Контрагент` | СправочникСсылка (будь-який), **вибір груп і елементів** | доступні клієнти — елемент `Контрагенты` або ціла **група** |
| ТЧ `Товары.Номенклатура` | СправочникСсылка (будь-який), **вибір груп і елементів** | доступна номенклатура — елемент `Номенклатура` або ціла **група** |

### Узагальнені типи реквізитів (важливо)
Реквізити-посилання задано **узагальненим типом `СправочникСсылка`** (`<v8:TypeSet>cfg:CatalogRef</v8:TypeSet>`), а **не** `CatalogRef.Склады` тощо. Причина: інакше XML не завантажується standalone — `Невідоме ім'я типу CatalogRef.Склады` (типів УТ ще немає до об'єднання).

Наслідки:
- Узагальнення потрібне **лише в реквізиті метаданих** — щоб не ламалося створення/завантаження CF. У **запитах типізація не потрібна**: значення в даних — справжні посилання (`Склады`, `Номенклатура`…), а вибірки товарів/контрагентів ідуть `ИЗ Справочник.Номенклатура/Контрагенты` (конкретний довідник), тож доступ до полів через крапку вже типізований. `ВЫРАЗИТЬ` у запитах не використовується.
- Код, що читає `Устройство.Склад` тощо в BSL, працює без змін (значення конкретні).
- ⚠️ У формі елемента поле дає вибір із будь-якого довідника — оператор має обрати правильний. Звуження вибору в UI (форма з обробником вибору) — окрема опціональна доробка.

### Вибір груп у ТЧ «Контрагенты» / «Товары»
Реквізити цих ТЧ задано з `ChoiceFoldersAndItems = FoldersAndItems` — оператор може
покласти в список **або окремий елемент, або групу** (папку) довідника. У запитах
`Module.bsl` обраний набір **розгортається в листові елементи** через
`… В ИЕРАРХИИ (ВЫБРАТЬ … ИЗ ВТ_Выбор) И НЕ …ЭтоГруппа`:
- обрана **група** → усі товари/контрагенти всередині неї (на будь-якій глибині, включно з **підгрупами**);
- обраний **елемент** → лише він.

**Без повторів.** Навіть якщо в ТЧ вказати групу + її підгрупу + окремий елемент,
що перетинаються, кожна номенклатура/контрагент потрапляє в результат **один раз**:
вибірка розгорнутих елементів іде `ИЗ` самого довідника (`РАЗЛИЧНЫЕ`), а регістри
(ціни, залишки, резерв, борг) попередньо **згорнуто по номенклатурі/контрагенту**
окремими тимчасовими таблицями — фінальні з'єднання строго 1:1, без декартового
розмноження за вимірами характеристик/упаковок/договорів.

> Передумова: довідники `Номенклатура` і `Контрагенты` в УТ — ієрархія **груп і
> елементів** (мають `ЭтоГруппа`). Якщо в цільовій базі котрийсь із них — ієрархія
> лише елементів, прибрати для нього `НЕ …ЭтоГруппа`.

### Ієрархія номенклатури (довільна вкладеність, ROADMAP #31)
`GET /categories` віддає **повне дерево груп** доступної номенклатури, а не один
рівень. Кожен вузол:

| Поле | Опис |
|---|---|
| `id` | GUID групи |
| `name` | назва групи |
| `parentId` | GUID батьківської групи; **`""` → коренева** (звідси будується дерево) |
| `count` | к-сть товарів **безпосередньо** в групі (проміжні групи → 0) |
| `expanded` | для UI-акордеону |

Кожен товар у `GET /products` несе `categoryId` (GUID безпосередньої групи) для
прив'язки до вузла дерева. Групи-предки збираються підйомом по `Родитель` в
`КатегорииУстройства` (ітеративно, бо в мові запитів 1С немає рекурсії; захист
від зациклення — 50 рівнів).

## HTTP-сервіс (`венд_МобильноеПриложение`, rootURL `vendo`)

Контракт **1:1** з демо-бекендом. Пристрій передається заголовком `X-Device-Id`
або параметром `?device=<GUID>`.

| Метод | Шлях | Опис |
|---|---|---|
| POST | `/auth` | `{deviceId}` → валідація пристрою, користувач + токен |
| GET | `/products` | номенклатура пристрою з ціною і залишком (склад − резерв) |
| GET | `/categories` | групи номенклатури пристрою |
| GET | `/customers` | контрагенти пристрою з боргом |
| GET | `/orders?startDate=&endDate=` | замовлення менеджера за період |
| POST | `/orders` | створити `ЗаказПокупателя` |
| PUT | `/orders/{num}` | оновити склад/контрагента |
| DELETE | `/orders/{num}` | видалити замовлення |

### Локалізація повідомлень (issue #26)
Повідомлення сервісу віддаються 3 мовами (uk / ru / en) через штатну функцію
**`НСтр`**: `НСтр("uk='…'; ru='…'; en='…'", КодЯзыкаЗапроса(Запрос))`. Мова
визначається з заголовка **`Accept-Language`** (або параметра `?lang=`); якщо не
розпізнано — **англійська** як запасна. Реалізація в `Module.bsl`:
- `КодЯзыкаЗапроса(Запрос)` — повертає `uk` / `ru` / `en` за первинним субтегом
  (`uk-UA` → `uk`, `ru-RU` → `ru`, інше → `en`); цей код передається другим
  аргументом у `НСтр`, тож мова береться із запиту, а не із сесії.

Локалізовано всі повідомлення помилок/статусів відповідей (`/auth`, не авторизовано,
некоректне тіло, замовлення не знайдено/видалено).

> **Поза бекендом (решта #26):** локалізація UI (i18next/react-intl), формати
> дат/чисел/валюти, ручне перемикання мови — фронтендна частина. Бекенд тут лише
> віддає локалізовані **повідомлення**; **статуси замовлень** (`Чернетка`/`Відправлено`)
> навмисно лишені канонічними значеннями — їх локалізує фронтенд як мітки, щоб не
> ламати звірку статусів.

## Перевірено на бойовій базі (17.06.2026)

Сервіс опубліковано (`https://<сервер>/<база>/hs/vendo`), basic-auth,
пристрій `000000001`:
- ✅ `/auth`, `/categories` (ієрархія), `/products` (реальні залишки), `/orders`;
- ✅ локалізація `Accept-Language`/`?lang` (uk/ru/en);
- 🔧 `/customers` падав 500 — `Контрагенты` **не має** `ЮридическийАдрес` (адреса — в
  регістрі `КонтактнаяИнформация`). Виправлено + реалізовано **реальну адресу й телефон**
  з `КонтактнаяИнформация` (за `Тип = Адрес/Телефон`) і ім'я **контактної особи** з
  `ОсновноеКонтактноеЛицо`;
- ⚠️ `price: 0` у всіх товарах — це **дані**: у пристрою не заповнено `ТипЦен` або
  немає записів `ЦеныНоменклатуры` для цього типу (запит коректний). Перевірити запис пристрою.

### Контактна інформація контрагента (`/customers`)
Адреса, телефон і контактна особа беруться так:
- **адреса** (`address`, дублюється в `city` для поточного UI) — `КонтактнаяИнформация`,
  `Тип = Адрес`, ресурс `Представление`;
- **телефон** (`phone`) — те саме, `Тип = Телефон`;
- **контактна особа** (`contact`) — реквізит `Контрагенты.ОсновноеКонтактноеЛицо`.

`Представление` — необмежений рядок, тому для агрегації кастується `ВЫРАЗИТЬ(… КАК
СТРОКА(n))`. На контрагента береться одне значення (`МАКСИМУМ`). Телефон контактної
особи поки не тягнемо (можливе розширення — `КонтактнаяИнформация` за `Объект` = особа).
