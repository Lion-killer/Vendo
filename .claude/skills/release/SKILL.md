# Випуск релізу Vendo

Оркеструєш повний цикл релізу. Механічну частину робить `frontend/scripts/release.mjs`
(авто-тип, версія, CHANGELOG.md, підписаний APK, GitHub Release) — твоя робота:
перевірки, редакторське оновлення документації і фінальний звіт.

## Кроки

### 1. Передумови (зупинись і повідом, якщо не так)
- Гілка `main`, робоче дерево чисте, локальна гілка не відстає від origin.
- Є коміти від останнього тега: `git log $(git describe --tags --abbrev=0)..HEAD --oneline`
  (якщо тегів ще немає — беруться всі коміти). Порожньо → нічого релізити, зупинись.
- **Перевірки зелені** (той самий гейт, що й CI — щоб не запушити червоний main):
  `npm run lint` (з кореня репо — oxlint по frontend + backend/mock) і
  `cd frontend && npm test`. Є попередження/падіння → зупинись, повідом, полагодь.

### 2. Документація (редакторський крок — до збірки!)
Переглянь коміти від останнього тега. Для кожної **user-facing** зміни (нова
можливість, змінена поведінка екрана, нове налаштування):
- онови відповідний розділ `docs/user-guide/*.md` (вшита довідка збирається в APK
  саме звідси — застаріла довідка потрапить у реліз назавжди);
- якщо зміна потребує нового скриншота — зроби, або якщо неможливо, попередь
  користувача і запитай, чи продовжувати без нього;
- закоміть правки доків (`docs(user-guide): …`) перед збіркою.

Технічні коміти (refactor/chore/ci) правок доків не потребують.

### 3. Погодження чейнджлогу (обовʼязково — до релізу!)
```bash
cd frontend && node scripts/changelog.mjs --draft
```
Покажи чернетку користувачу в чаті й **чекай явного погодження**:
- «ок» без правок → запиши чернетку як є у `frontend/.changelog-draft.md`;
- правки → запиши **відредагований** текст у `frontend/.changelog-draft.md`.

Файл підхопиться при релізі (він, а не автогенерація, піде в CHANGELOG.md і в
нотатки GitHub-релізу) і зітреться автоматично. **Без погодження реліз не запускати.**

### 4. Реліз
```bash
cd frontend && npm run release          # auto: !/BREAKING → major, feat → minor, інакше patch
```
Скрипт сам: визначить тип → `npm version` (bump + sync Android + CHANGELOG.md + тег)
→ підписаний APK → push → GitHub Release з нотатками з CHANGELOG.

Явний тип, якщо користувач попросив: `npm run release -- minor`.

**Перший реліз на машині:** скрипт згенерує keystore — обовʼязково передай
користувачу попередження зробити бекап `android/vendo-release.keystore` і
`android/keystore.properties` (втрата = оновлення не встановляться поверх старих APK).

### 5. Звіт
- Посилання на реліз, версія, тип (major/minor/patch) і чому.
- Що оновлено в доках (або «доки не потребували змін»).
- Нагадай: користувачі побачать кнопку оновлення в меню профілю за сесію.

## Заборонено
- Релізити з брудного дерева чи не з main.
- Релізити з червоним лінтом/тестами (крок 1) — CI все одно відхилить push.
- Пропускати крок документації «бо дрібниця» — рішення пропустити ухвалює користувач.
- Публікувати без погодженого користувачем чейнджлогу (крок 3).
- `--force`-пуш тегів; тег уже існує → зупинись і спитай.
