# Мок-бекенд Vendo (Node + Express, in-memory)

Демонстраційна реалізація **єдиного REST-контракту** для локальної розробки й тесту
фронтенду — без 1С. Авторизація **імітована**, дані — із фікстури `data/db.json`,
зберігаються **в пам'яті**. Справжній бекенд — 1С HTTP-сервіс (`../1c-config/TradeUkr23/`).

## Запуск

```bash
npm install
npm start          # http://localhost:3000, базовий шлях /api
```

- Swagger UI контракту: http://localhost:3000/api/docs (спека — `openapi.json`).
- Немає watch/reload — після правок перезапусти вручну (`require` кешує модулі).
- Жодних нативних модулів/збірки — чистий Node (`express` + `cors`).

## Тести

```bash
npm test           # node --test (без залежностей); чисті хелпери lib/orders.js
```

## Дані

- **Сід при кожному старті** з `data/db.json` у пам'ять (`db.js`).
- Мутації (замовлення) живуть **у пам'яті**; на рестарті — чистий сід (мок навмисно
  не персистить, тож завжди відома відправна точка). Немає файлів БД для очищення.

## Ендпоінти

| Метод | Шлях | Опис |
|---|---|---|
| `GET`/`HEAD` | `/health` | дешева перевірка доступності (без авторизації) |
| `POST` | `/auth` | **стаб**: фіксований користувач + `mock_token_123` (нічого не перевіряє) |
| `GET` | `/products` | номенклатура (ціна, залишок, штрихкод, фото) |
| `GET` | `/products/{id}/image` | завжди 404 (демо не зберігає бінарних фото — як 1С без зображення) |
| `GET` | `/categories` | категорії (дерево, `parentId`) |
| `GET` | `/customers` | контрагенти (адреса/телефон/контактні особи/борг) |
| `GET` | `/orders?startDate=&endDate=` | замовлення за період |
| `POST` | `/orders` | upsert за клієнтським GUID (ідемпотентно; конфлікт версій → 409) |
| `PUT` | `/orders/{id}` | оновити замовлення (за GUID) |
| `DELETE` | `/orders/{id}` | «Нове» — видалити; інше — помітка на видалення |

## Чим відрізняється від 1С

Мок реалізує **той самий контракт**, але як заглушка:
- `POST /auth` — **стаб**: не перевіряє код прив'язки, завжди повертає токен.
- **Не скоупить за пристроєм** — віддає всі сід-дані (1С фільтрує за `X-Device-Id`:
  склад / тип цін / менеджер / контрагенти / товари).
- Заголовки `X-Device-Id` / `X-Auth-Token` **не перевіряються**.
- Зберігання — в пам'яті (1С — у документах УТ); персистентності між рестартами немає.

## Контракт

Канон — [`openapi.json`](openapi.json) (Swagger на `/api/docs`). Замовлення зберігаються
**нормалізовано** (посилання на товар/контрагента + зафіксована ціна per-рядок — snapshot
на момент замовлення), а на віддачі **гідратуються** (`hydrateOrder` добудовує
`customer`/`items[].product`/`total`). Номери — з лічильника (`ЗМ-N`, не повторюються).
Оптимістична конкуренція — токен `version` (як `ВерсияДанных` у 1С): редагування від
застарілої версії → `409`. Будь-яка інша реалізація контракту (зокрема 1С) сумісна без змін.
