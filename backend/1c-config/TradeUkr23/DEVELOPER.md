# Міні-конфігурація 1С Vendo — для розробника

Технічна частина до [README.md](README.md) (функціонал і встановлення). Тут —
реалізація: структура, типи реквізитів, запити, локалізація, BSL-нюанси.

## Склад

```
src/
├─ Configuration.xml                       # маніфест конфігурації-постачальника
├─ Catalogs/венд_МобильныеУстройства.xml   # довідник «Мобільні пристрої»
├─ HTTPServices/венд_МобильноеПриложение/  # HTTP-сервіс (REST API)
│  └─ Ext/Module.bsl                        # обробники методів
├─ Roles/венд_ОсновнаяРоль/Ext/Rights.xml  # права на власні об'єкти
└─ Languages/{English,Русский,Українська}.xml
```

## ⚠️ Код модуля компілюється лише в УТ

Метадані вантажаться в **пусту базу** (реквізити-посилання задано узагальненим типом
`СправочникСсылка`, див. нижче), тож із них і роблять CF. Але **код** `Module.bsl`
звертається до типів цільової УТ — довідники `Склады` / `Контрагенты` / `Номенклатура` /
`ТипыЦенНоменклатуры` / `Пользователи`, `Документ.ЗаказПокупателя`, регістри
`ТоварыНаСкладах` / `ТоварыВРезервеНаСкладах` / `ЦеныНоменклатуры` /
`ВзаиморасчетыСКонтрагентами`. Їх немає в пустій базі, тому код **компілюється й працює
лише після об'єднання з УТ**. `cf-validate` тут перевіряє лише структуру власних об'єктів.

## Узагальнені типи реквізитів

Реквізити-посилання довідника «Мобільні пристрої» задано **узагальненим типом
`СправочникСсылка`** (`<v8:TypeSet>cfg:CatalogRef</v8:TypeSet>`), а **не**
`CatalogRef.Склады` тощо. Причина: інакше XML не завантажується standalone —
`Невідоме ім'я типу CatalogRef.Склады` (типів УТ ще немає до об'єднання).

Наслідки:

- Узагальнення потрібне **лише в реквізиті метаданих** — щоб не ламалося
  створення/завантаження CF. У **запитах типізація не потрібна**: значення в даних —
  справжні посилання (`Склады`, `Номенклатура`…), а вибірки товарів/контрагентів ідуть
  `ИЗ Справочник.Номенклатура/Контрагенты` (конкретний довідник), тож доступ до полів
  через крапку вже типізований. `ВЫРАЗИТЬ` у запитах не використовується.
- Код, що читає `Устройство.Склад` тощо в BSL, працює без змін (значення конкретні).
- У **формі елемента** тип кожного поля **звужено програмно** до конкретного довідника
  УТ (`ПриСозданииНаСервере` → `ОграничитьТипыСсылок`: `ОграничениеТипа` через рядкове
  `ОписаниеТипов`): Склад→`Склады`, ТипЦен→`ТипыЦенНоменклатуры`, Менеджер→`Пользователи`,
  контрагент→`Контрагенты`, номенклатура→`Номенклатура`. У standalone (без УТ) обмеження
  тихо пропускається; у злитій конфігурації тип резолвиться — оператор обирає лише з
  правильного довідника.

## Розгортання вибору груп у ТЧ «Контрагенты» / «Товары»

Реквізити цих ТЧ задано з `ChoiceFoldersAndItems = FoldersAndItems` — оператор може
покласти в список **або окремий елемент, або групу** (папку) довідника. У запитах
`Module.bsl` обраний набір **розгортається в листові елементи** через
`… В ИЕРАРХИИ (ВЫБРАТЬ … ИЗ ВТ_Выбор) И НЕ …ЭтоГруппа`:

- обрана **група** → усі товари/контрагенти всередині неї (на будь-якій глибині, включно
  з **підгрупами**);
- обраний **елемент** → лише він.

**Без повторів.** Навіть якщо в ТЧ вказати групу + її підгрупу + окремий елемент,
що перетинаються, кожна номенклатура/контрагент потрапляє в результат **один раз**:
вибірка розгорнутих елементів іде `ИЗ` самого довідника (`РАЗЛИЧНЫЕ`), а регістри
(ціни, залишки, резерв, борг) попередньо **згорнуто по номенклатурі/контрагенту**
окремими тимчасовими таблицями — фінальні з'єднання строго 1:1, без декартового
розмноження за вимірами характеристик/упаковок/договорів.

> Передумова: довідники `Номенклатура` і `Контрагенты` в УТ — ієрархія **груп і
> елементів** (мають `ЭтоГруппа`). Якщо в цільовій базі котрийсь із них — ієрархія
> лише елементів, прибрати для нього `НЕ …ЭтоГруппа`.

## Ієрархія номенклатури (`GET /categories`, ROADMAP #31)

Віддає **повне дерево груп** доступної номенклатури, а не один рівень. Кожен вузол:

| Поле | Опис |
|---|---|
| `id` | GUID групи |
| `name` | назва групи |
| `parentId` | GUID батьківської групи; **`""` → коренева** (звідси будується дерево) |
| `count` | к-сть товарів **безпосередньо** в групі (проміжні групи → 0) |
| `expanded` | для UI-акордеону |

Кожен товар у `GET /products` несе `categoryId` (GUID безпосередньої групи) для
прив'язки до вузла дерева. Групи-предки збираються підйомом по `Родитель` в
`КатегорииУстройства` (ітеративно, бо в мові запитів 1С немає рекурсії; захист
від зациклення — 50 рівнів).

## Маршрутизація замовлень

За **GUID-`id`** (`НайтиЗаказПоИд`), не за номером документа. Гідратоване замовлення
несе фото позицій тим самим шляхом `/products/{id}/image?v=`, що й каталог.

## Локалізація повідомлень (issue #26)

Повідомлення сервісу віддаються 3 мовами (uk / ru / en) через штатну функцію
**`НСтр`**: `НСтр("uk='…'; ru='…'; en='…'", КодЯзыкаЗапроса(Запрос))`. Мова
визначається з заголовка **`Accept-Language`** (або параметра `?lang=`); якщо не
розпізнано — **англійська** як запасна. Реалізація в `Module.bsl`:

- `КодЯзыкаЗапроса(Запрос)` — повертає `uk` / `ru` / `en` за первинним субтегом
  (`uk-UA` → `uk`, `ru-RU` → `ru`, інше → `en`); цей код передається другим
  аргументом у `НСтр`, тож мова береться із запиту, а не із сесії.

Локалізовано всі повідомлення помилок/статусів відповідей (`/auth`, не авторизовано,
некоректне тіло, замовлення не знайдено/видалено).

> **Поза бекендом (решта #26):** локалізація UI (i18next/react-intl), формати
> дат/чисел/валюти, ручне перемикання мови — фронтендна частина. Бекенд тут лише
> віддає локалізовані **повідомлення**; **статуси замовлень** (`Чернетка`/`Відправлено`)
> навмисно лишені канонічними значеннями — їх локалізує фронтенд як мітки, щоб не
> ламати звірку статусів.

## Контактна інформація контрагента (`/customers`)

Адреса, телефон і контактна особа беруться так:

- **адреса** (`address`, дублюється в `city` для поточного UI) — `КонтактнаяИнформация`,
  `Тип = Адрес`, ресурс `Представление`;
- **телефон** (`phone`) — те саме, `Тип = Телефон`;
- **контактна особа** (`contact`) — реквізит `Контрагенты.ОсновноеКонтактноеЛицо`.

`Представление` — необмежений рядок, тому для агрегації кастується `ВЫРАЗИТЬ(… КАК
СТРОКА(n))`. На контрагента береться одне значення (`МАКСИМУМ`). Телефон контактної
особи поки не тягнемо (можливе розширення — `КонтактнаяИнформация` за `Объект` = особа).

## Потребує вивірки в цільовій УТ

Створення `ЗаказПокупателя` використовує специфічні реквізити (позначено `// TODO`
в `Module.bsl`):

- **Организация** — зараз перша з довідника; задати правило вибору.
- **ДоговорКонтрагента** — `ОсновнойДоговорВзаиморасчетов`; може бути обов'язковим.
- **Валюта/НДС** — за потреби заповнити явно.

**Проведення** документа — **реалізовано** (`ЗаписатьЗаказ`→`НужноПровести`): за прапорцем
пристрою `АвтоПроведение` (або явним статусом «Проведено» в тілі) запис іде
`Записать(РежимЗаписиДокумента.Проведение)` з рухами, інакше `Записать(Запись)`. Запис і
проведення — в одній транзакції. Потребує перевірки в цільовій УТ: контроль залишків чи
незаповнені обов'язкові поля можуть блокувати проведення (тоді `POST`/`PUT` → 400 з описом).
