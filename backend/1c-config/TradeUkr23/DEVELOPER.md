# Міні-конфігурація 1С Vendo — для розробника

Технічна частина до [README.md](README.md) (довідник, QR, прив'язка — для операторів УТ).
Тут — встановлення, контракт API і реалізація: типи реквізитів, запити, прив'язка,
локалізація, BSL-нюанси.

Окрема **конфігурація-постачальник** для УТ для України 2.3, яку **вбудовують у цільову
УТ через «Порівняння/об'єднання конфігурацій»**.

- Префікс об'єктів: `венд_`
- Сумісність: `Version8_3_19` — працює навіть у режимі сумісності **8.2.13**
  (HTTP-сервіс вимагає лише платформи **≥ 8.3.5**)

## Склад

```
src/
├─ Configuration.xml                       # маніфест конфігурації-постачальника
├─ Catalogs/венд_МобильныеУстройства.xml   # довідник «Мобільні пристрої» (+ макет ГенераторQR)
│  └─ Ext/ManagerModule.bsl                 # доступ до ХранилищеОбщихНастроек (URL/токен/код/лог)
├─ HTTPServices/венд_МобильноеПриложение/  # HTTP-сервіс (REST API)
│  └─ Ext/Module.bsl                        # обробники методів
├─ Roles/венд_HTTPСервис/                  # єдина роль: Use сервісу для користувача публікації
└─ Languages/{English,Русский,Українська}.xml
```

## Версія міні-конфігурації

Властивість **Version** у `Configuration.xml` синхронізується з версією релізу Vendo
автоматично (`frontend/scripts/sync-version.mjs` при кожному релізі). У Конфігураторі
(властивості конфігурації → Версія) видно, з яким релізом додатка узгоджений бекенд —
руками не редагувати.

## Як вбудувати в цільову УТ

1. Завантажити конфігурацію з XML-файлів (`LoadConfigFromFiles`) у **пусту базу**.
2. Зберегти конфігурацію у **CF**-файл (Конфігуратор → «Зберегти конфігурацію у файл»).
3. У базі цільової УТ **об'єднати** цей CF (**Порівняння й об'єднання конфігурацій** →
   перенести `венд_*` об'єкти).
4. Оновити конфігурацію БД, опублікувати HTTP-сервіс на веб-сервері (Apache/IIS) —
   отримати URL виду `https://<сервер>/<база>/hs/vendo`.
5. Завести записи довідника «Мобільні пристрої» (див. [README.md](README.md)).

Адресу опублікованого сервісу задає вже оператор — командою **«Налаштувати URL сервісу»**
зі списку «Мобільні пристрої» (пише у `ХранилищеОбщихНастроек`, неймспейс `vendo`, без
Конфігуратора). Звідти вона зашивається в QR.

Об'єднання працює навіть якщо УТ у режимі сумісності 8.2.13 — обмежень немає.

## ⚠️ Код модуля компілюється лише в УТ

Метадані вантажаться в **пусту базу** (реквізити-посилання задано узагальненим типом
`СправочникСсылка`, див. нижче), тож із них і роблять CF. Але **код** `Module.bsl`
звертається до типів цільової УТ — довідники `Склады` / `Контрагенты` / `Номенклатура` /
`ТипыЦенНоменклатуры` / `Пользователи`, `Документ.ЗаказПокупателя`, регістри
`ТоварыНаСкладах` / `ТоварыВРезервеНаСкладах` / `ЦеныНоменклатуры` /
`ВзаиморасчетыСКонтрагентами`. Їх немає в пустій базі, тому код **компілюється й працює
лише після об'єднання з УТ**. `cf-validate` тут перевіряє лише структуру власних об'єктів.

## Узагальнені типи реквізитів

Реквізити-посилання довідника «Мобільні пристрої» задано **узагальненим типом
`СправочникСсылка`** (`<v8:TypeSet>cfg:CatalogRef</v8:TypeSet>`), а **не**
`CatalogRef.Склады` тощо. Причина: інакше XML не завантажується standalone —
`Невідоме ім'я типу CatalogRef.Склады` (типів УТ ще немає до об'єднання).

Наслідки:

- Узагальнення потрібне **лише в реквізиті метаданих** — щоб не ламалося
  створення/завантаження CF. У **запитах типізація не потрібна**: значення в даних —
  справжні посилання (`Склады`, `Номенклатура`…), а вибірки товарів/контрагентів ідуть
  `ИЗ Справочник.Номенклатура/Контрагенты` (конкретний довідник), тож доступ до полів
  через крапку вже типізований. `ВЫРАЗИТЬ` у запитах не використовується.
- Код, що читає `Устройство.Склад` тощо в BSL, працює без змін (значення конкретні).
- У **формі елемента** тип кожного поля **звужено програмно** до конкретного довідника
  УТ (`ПриСозданииНаСервере` → `ОграничитьТипыСсылок`: `ОграничениеТипа` через рядкове
  `ОписаниеТипов`): Склад→`Склады`, ТипЦен→`ТипыЦенНоменклатуры`, Менеджер→`Пользователи`,
  контрагент→`Контрагенты`, номенклатура→`Номенклатура`. У standalone (без УТ) обмеження
  тихо пропускається; у злитій конфігурації тип резолвиться — оператор обирає лише з
  правильного довідника.

## Розгортання вибору груп у ТЧ «Контрагенты» / «Товары»

Реквізити цих ТЧ задано з `ChoiceFoldersAndItems = FoldersAndItems` — оператор може
покласти в список **або окремий елемент, або групу** (папку) довідника. У запитах
`Module.bsl` обраний набір **розгортається в листові елементи** через
`… В ИЕРАРХИИ (ВЫБРАТЬ … ИЗ ВТ_Выбор) И НЕ …ЭтоГруппа`:

- обрана **група** → усі товари/контрагенти всередині неї (на будь-якій глибині, включно
  з **підгрупами**);
- обраний **елемент** → лише він.

**Без повторів.** Навіть якщо в ТЧ вказати групу + її підгрупу + окремий елемент,
що перетинаються, кожна номенклатура/контрагент потрапляє в результат **один раз**:
вибірка розгорнутих елементів іде `ИЗ` самого довідника (`РАЗЛИЧНЫЕ`), а регістри
(ціни, залишки, резерв, борг) попередньо **згорнуто по номенклатурі/контрагенту**
окремими тимчасовими таблицями — фінальні з'єднання строго 1:1, без декартового
розмноження за вимірами характеристик/упаковок/договорів.

> Передумова: довідники `Номенклатура` і `Контрагенты` в УТ — ієрархія **груп і
> елементів** (мають `ЭтоГруппа`). Якщо в цільовій базі котрийсь із них — ієрархія
> лише елементів, прибрати для нього `НЕ …ЭтоГруппа`.

## Ієрархія номенклатури (`GET /categories`)

Віддає **повне дерево груп** доступної номенклатури, а не один рівень. Кожен вузол:

| Поле | Опис |
|---|---|
| `id` | GUID групи |
| `name` | назва групи |
| `parentId` | GUID батьківської групи; **`""` → коренева** (звідси будується дерево) |
| `count` | к-сть товарів **безпосередньо** в групі (проміжні групи → 0) |
| `expanded` | для UI-акордеону |

Кожен товар у `GET /products` несе `categoryId` (GUID безпосередньої групи) для
прив'язки до вузла дерева. Групи-предки збираються підйомом по `Родитель` в
`КатегорииУстройства` (ітеративно, бо в мові запитів 1С немає рекурсії; захист
від зациклення — 50 рівнів).

## HTTP-сервіс (`венд_МобильноеПриложение`, rootURL `vendo`)

Контракт **1:1** з демо-бекендом. Пристрій передається заголовком `X-Device-Id`
або параметром `?device=<GUID>`.

| Метод | Шлях | Опис |
|---|---|---|
| GET/HEAD | `/health` | дешева перевірка доступності (без авторизації) |
| POST | `/auth` | `{deviceId, pairingCode}` → валідація пристрою + обмін одноразового коду на bearer-токен |
| GET | `/products` | номенклатура пристрою з ціною, залишком (склад − резерв), штрихкодом, фото |
| GET | `/products/{id}/image` | бінарне фото товару (`ОсновноеИзображение`; `?v=` — інвалідація кешу) |
| GET | `/categories` | повне дерево груп номенклатури пристрою |
| GET | `/customers` | контрагенти пристрою з боргом, адресою, телефоном, контактними особами |
| GET | `/orders?startDate=&endDate=` | замовлення менеджера за період; глибина обмежена карткою пристрою (`ГлубинаИсторииЗаказов`, 0 = 30 днів) — початок періоду піднімається до капу незалежно від фільтра клієнта |
| POST | `/orders` | upsert `ЗаказПокупателя` за клієнтським GUID (ідемпотентно) |
| PUT | `/orders/{id}` | оновити замовлення (за GUID) |
| DELETE | `/orders/{id}` | помітка на видалення (за GUID) |
| POST | `/telemetry` | снапшот стану пристрою (+опційно лог) → запис регістра телеметрії; відповідь несе `requestLog` |

> Маршрутизація — за **GUID-`id`** (`НайтиЗаказПоИд`), не за номером документа. Гідратоване
> замовлення несе фото позицій тим самим шляхом `/products/{id}/image?v=`, що й каталог.

> ⚠️ **Серіалізація замовлень — лише пакетним запитом** (`ВыборкаЗаказов`: шапка +
> вкладена ТЧ `Заказ.Товары.(…)` з реквізитами номенклатури через крапку). Викликач
> готує `ВТ_Заказы` (посилання) у своєму `МенеджерВременныхТаблиц` — список фільтрами,
> одиночне читання одним `&Ссылка`; SELECT-літерал один на всіх, без склейки. НІЯКИХ
> `ПолучитьОбъект`/розіменувань посилань у циклі — подокументна гідратація на повній
> історії менеджера займала ~60 с і 42 МБ (замір на бойовій), пакетна — секунди. Поза
> запитом читається лише версія зображення (`ПутьИзображения`, кешується); навіть
> `ВерсияДанных` береться з запиту (доступна в мові запитів з 8.3.11).

> ⚠️ **Заголовки читати ТІЛЬКИ через `ЗаголовокЗапроса(Запрос, Имя)`** (регістронезалежний
> пошук), не через `Запрос.Заголовки.Получить()`. `Заголовки` — звичайне Соответствие з
> регістрозалежним пошуком, а HTTP/2 і нативний шар Capacitor передають імена заголовків
> у нижньому регістрі (`x-device-id`) — пряме `Получить("X-Device-Id")` їх не бачить, і
> валідні запити пристроїв відбивалися 401 (живий інцидент із `POST /telemetry`).

### Обробка помилок: стандартний формат, без виключень

Будь-яка неуспішна відповідь сервісу — **завжди** JSON `{success: false, message}`
з кодом стану; інших форматів помилок немає.

- **Очікувані відмови** (не вказано/не знайдено контрагента чи товар, некоректне
  тіло, замовлення не знайдено, проведене не можна редагувати/видаляти, проведення
  не пройшло) — обробляються **без виключень**: `СохранитьЗаказ` (єдиний шлях
  створення І оновлення)/`ЗаполнитьТовары`/`ЗаписатьЗаказ` повертають ознаку успіху
  + `ТекстОшибки`,
  обробник віддає `ОтветОшибки(Сообщение, Код)` → 4xx. Кожна така відмова пишеться
  в **журнал реєстрації** («венд_МобильноеПриложение», Предупреждение) — причини
  відмов синхронізації видно без журналу пристрою.
- **Помилки проведення**: текст для клієнта збирається з повідомлень користувачу
  (`ПолучитьСообщенияПользователю` — «Не хватает остатка…»), а не зі службового
  тексту виключення платформи.
- **Неочікувані збої** — єдине місце для виключень: зовнішня `Попытка` обробника →
  `ОтветИсключение` → журнал (Ошибка) + 500 у тому ж JSON-форматі.

> ⚠️ **IIS підміняє тіла помилок.** Без налаштування custom error pages IIS замінює
> JSON-тіло відповідей 401/404/500 своєю HTML-сторінкою (`Страница не найдена…`,
> windows-1251) — додаток бачить `Failed to execute 'json' on 'Response'` замість
> повідомлення. У `web.config` **папки публікації** (поруч із `default.vrd`) має бути:
>
> ```xml
> <configuration>
>   <system.webServer>
>     <httpErrors existingResponse="PassThrough" />
>   </system.webServer>
> </configuration>
> ```
>
> Якщо `web.config` уже існує — додати лише рядок `<httpErrors …/>` у
> `<system.webServer>`. Після зміни — `iisreset` не потрібен, конфіг підхоплюється.

## Прив'язка пристрою (QR → код → токен)

Ідентичність пристрою — реквізит `ИдентификаторУстройства` (GUID = посилання елемента,
`УстановитьСсылкуНового`; read-only після запису) + `Заблокировано`. **Мутабельний
рантайм-стан** (код прив'язки, хеш токена, прапорець запиту логу, URL сервісу) — **НЕ на
елементі**, а у `ХранилищеОбщихНастроек` під фіксованим користувачем **`vendo`**. Мета:
**API ніколи не пише елемент довідника** — той лишається чистою конфігурацією, яку редагує
тільки оператор (нема churn `ВерсияДанных`, блокувань, конфліктів із карткою).

- **Доступ до сховища** — модуль менеджера `Справочники.венд_МобильныеУстройства`:
  `АдресСервиса`/`УстановитьАдресСервиса`, `ХешТокена`/`УстановитьХешТокена`,
  `КодПривязки`/`УстановитьКодПривязки`, `ЗапрошенЛог`/`УстановитьЗапросЛога`,
  `ОчиститьСостояние`. Усі — привілейований режим, неймспейс `vendo` (сесія сервісу — це
  `vendo`; форми оператора йдуть у той самий неймспейс явно). Ключі:
  `венд_Сервис/URL`, `венд_ТокенУстройства/<id>`, `венд_КодПривязки/<id>`,
  `венд_ЗапроситьЛог/<id>`.
- **QR** малюється у полі HTML-документа форми (ця стара БСП не має
  `УправлениеПечатью.ДанныеQRКода`): бібліотека `qrcode.js` **вшита** в self-contained
  HTML (макет `ГенераторQR`), рядок присвоюється реквізиту форми **на сервері** — лише так
  платформа виконує `<script>` усередині поля HTML. Вміст QR — JSON
  `{ deviceId, apiUrl, code }`, де `apiUrl` і `code` — зі сховища.
- **`/auth`** перевіряє `Заблокировано`, звіряє `pairingCode` ↔ код прив'язки зі сховища
  (без регістру), видає токен і зберігає у сховищі `Base64(SHA-256(токен))` (сам токен
  не зберігається), а код прив'язки **гасить** — одноразовий. Без `ПолучитьОбъект/Записать`.
- **Кожен запит** перевіряє `Заблокировано` і `SHA-256(пред'явлений токен)` = хеш зі сховища.
- Форма елемента: код прив'язки — **реквізит форми**, вантажиться зі сховища при відкритті.
  `ПослеЗаписиНаСервере` генерує код **лише для нового, ще НЕ спареного** пристрою (нема ні
  коду, ні хешу токена) — щоб після авторизації повторний запис картки не відроджував код.
  Поки картка відкрита, обробник очікування (10 с) перечитує стан: після прив'язки (сервіс
  гасить код) код і QR **зникають з форми самі** + сповіщення. Кнопка **«Згенерувати код
  прив'язки»** — новий код + очищення токена у сховищі (переприв'язка), елемент не пишеться.

> **Деплой цієї зміни — переприв'язка.** Хеш токена раніше спарених пристроїв був на
> реквізиті (тепер видаленому) → після оновлення всі пристрої попросять пересканувати QR.
> URL сервісу теж задати заново (константу видалено).

## Телеметрія (#42)

Один **періодичний** (секунда) регістр відомостей **`венд_ТелеметрияУстройств`**:
вимір `Устройство`; ресурси `ПоследняяАктивность`, `Версия`, `Модель`,
`ПоследняяСинхронизация`, `Очередь`, `Ошибки`, `СетевыеТаймауты`, `Лог`
(`ХранилищеЗначения`).

- **`POST /telemetry`** (`TelemetryPost` → `ЗаписатьТелеметрию`): кожен снапшот —
  **новий запис** (історія станів). Поле `log` — плоский текст журналу додатка;
  стискається **на боці 1С** (`Новый ХранилищеЗначения(Текст, Новый СжатиеДанных(9))`),
  клієнт zip не робить. `Ошибки` (`logErrors`) — к-сть нових помилок **додатка** саме
  цього відвантаження (йде лише разом із логом).
- **`СетевыеТаймауты`** (`netErrors`) — таймаути/збої мережі за вікно від попереднього
  снапшота; йде **кожним** снапшотом (не лише з логом). Свідомо окремо від `Ошибки`:
  мережеві збої — штатний шум польового додатка (не червонити «Помилки» щодня), але
  стійко високі `СетевыеТаймауты` показують поганий канал/повільний сервер конкретного
  пристрою. Клієнт рахує їх за окремим маркером (`vendo_telemetry_net_mark`), тож це
  «за поточне вікно», а не накопичення за сесію.
- **Пасивна активність**: `ОпределитьУстройство` на кожен авторизований запит викликає
  `ОбновитьАктивностьУстройства` — тротльовано (5 хв) **перезаписує останній запис**
  (той самий період → історія не росте), оновлюючи активність і версію з заголовка
  `X-App-Version`. Помилки телеметрії ковтаються — вона не має ламати основний запит.
- **Запит логу** — кнопка **«Запросити лог»** у панелі **списку** пристроїв (для виділених
  ставить прапорець у `ХранилищеОбщихНастроек`, ключ `венд_ЗапроситьЛог/<id>` — не реквізит
  елемента). Стан видно в колонці **«Запит логу»**: поле-заглушка `ЛОЖЬ КАК ЗапросЛога` у
  запиті динсписку, значення підставляється зі сховища в події
  `СписокПриПолученииДанныхНаСервере`. Відповідь `/telemetry` повертає прапорець як
  `requestLog`; отримавши снапшот із `log`, сервіс **гасить його у сховищі**.
- **Форма списку пристроїв** — довільний запит динсписку (`ЛЕВОЕ СОЕДИНЕНИЕ` зі
  `СрезПоследних`): колонки активність/версія/модель/синхронізація/черга/помилки/таймаути +
  «Запит логу» (зі сховища через `СписокПриПолученииДанныхНаСервере`). Кнопки: «Запросити
  лог» (для виділених), «Телеметрія (журнал)» (список записів з відбором), «Очистити
  телеметрію (30+ днів)» (видаляє старі записи, лишаючи останній кожного пристрою),
  «Налаштувати URL сервісу».
- **Форма запису** регістра (`ФормаЗаписи`, за умовчанням) розпаковує `Лог` у текстове
  поле + «Зберегти лог у файл» (`ПолучитьФайл` із тимчасового сховища).

## Права: роль і користувач публікації

У міні-конфігурації **одна роль** — **`венд_HTTPСервис`**, для користувача, під
яким працює публікація HTTP-сервісу: **лише «Использование» на кожному методі**
шаблонів URL сервісу (12 методів; на самому вузлі HTTP-сервісу прав у платформи
немає — право визначене тільки для методів). Більше нічого не потрібно, бо…

> **Автостворення користувача.** Форма списку «Мобільні пристрої» при відкритті перевіряє
> `ПользователиИнформационнойБазы.НайтиПоИмени("vendo")` і, якщо його немає, пропонує
> створити (питає пароль — у код не зашитий, репозиторій публічний; має збігатися з
> публікацією). Створює IB-користувача зі стандартною автентифікацією + роль `венд_HTTPСервис`,
> прихований зі списку вибору. Потребує адмінправ (без них — повідомлення, форма не ламається).
> Це чистий `ПользователиИнформационнойБазы` без механізмів БСП.

### Привілейований режим замість прав на об'єкти УТ

Кожен обробник даних сервісу першим рядком вмикає
`УстановитьПривилегированныйРежим(Истина)` — увесь доступ до даних іде в обхід
прав 1С. Безпека тримається не на ролях, а на **перевірці токена пристрою**
(`ОпределитьУстройство`: пристрій не заблоковано + SHA-256 пред'явленого токена
збігається) і **скоупі пристрою** в усіх запитах (склад/тип цін/менеджер/
контрагенти/товари). `Health*` привілеїв не вмикають (даних не читають).

Переваги проти таблиці прав на об'єкти УТ: роль не розсипається при зміні
запитів/типових підписок (вони теж виконуються привілейовано), і мінімальна
роль справді мінімальна.

> ⚠️ **Роль і модуль — нероздільна пара.** Мінімальна роль працює лише з модулем,
> що вмикає привілейований режим. Якщо перенести в базу нову роль зі старим
> модулем — усі методи даних падають 500 «Порушення прав доступу» (перевірено
> на практиці). Переносити одним об'єднанням.

**Користувач публікації:** окремий користувач ІБ (напр. `ВендСервис`) лише з
роллю `венд_HTTPСервис`; ним підписується доступ публікації (`default.vrd`).
Інтерактивний вхід йому не потрібен (без ролей клієнтського запуску він і не
зможе — це плюс: викрадені креди публікації не дають зайти в базу клієнтом).

**Оператор** (ведення довідника «Мобільні пристрої», константа адреси сервісу)
працює під своїми правами УТ — окремої операторської ролі конфігурація не додає;
на практиці це користувач із повними правами (адміністратор бази).

## Локалізація повідомлень

Повідомлення сервісу віддаються 3 мовами (uk / ru / en) через штатну функцію
**`НСтр`**: `НСтр("uk='…'; ru='…'; en='…'", КодЯзыкаЗапроса(Запрос))`. Мова
визначається з заголовка **`Accept-Language`** (або параметра `?lang=`); якщо не
розпізнано — **англійська** як запасна. Реалізація в `Module.bsl`:

- `КодЯзыкаЗапроса(Запрос)` — повертає `uk` / `ru` / `en` за первинним субтегом
  (`uk-UA` → `uk`, `ru-RU` → `ru`, інше → `en`); цей код передається другим
  аргументом у `НСтр`, тож мова береться із запиту, а не із сесії.

Локалізовано всі повідомлення помилок/статусів відповідей (`/auth`, не авторизовано,
некоректне тіло, замовлення не знайдено/видалено).

> **Поза бекендом:** локалізація UI (i18next/react-intl), формати
> дат/чисел/валюти, ручне перемикання мови — фронтендна частина. Бекенд тут лише
> віддає локалізовані **повідомлення**; **статуси замовлень** (`Чернетка`/`Відправлено`)
> навмисно лишені канонічними значеннями — їх локалізує фронтенд як мітки, щоб не
> ламати звірку статусів.

## Контактна інформація контрагента (`/customers`)

Адреса, телефон і контактна особа беруться так:

- **адреса** (`address`, дублюється в `city` для поточного UI) — `КонтактнаяИнформация`,
  `Тип = Адрес`, ресурс `Представление`;
- **телефон** (`phone`) — те саме, `Тип = Телефон`;
- **контактна особа** (`contact`) — реквізит `Контрагенты.ОсновноеКонтактноеЛицо`.

`Представление` — необмежений рядок, тому для агрегації кастується `ВЫРАЗИТЬ(… КАК
СТРОКА(n))`. На контрагента береться одне значення (`МАКСИМУМ`). Телефон контактної
особи поки не тягнемо (можливе розширення — `КонтактнаяИнформация` за `Объект` = особа).

## Заповнення ЗаказПокупателя — типовими механізмами УТ

`СохранитьЗаказ` (єдина функція для створення і оновлення: `СсылкаЗаказа = Неопределено`
— нове, інакше часткова правка наявного) не заповнює документ вручну — використовується
той самий набір процедур, що й обміни з торговими майданчиками (зразок:
`Лис_ТоргПлощадки_ОбщегоНазначения.СоздатьНовыйЗаказ_Общее`):

- **Шапка** — `ЗаполнениеДокументов.ЗаполнитьШапкуДокумента(Заказ, ПользовательСеанса(),
  ВалютаДокумента, "Продажа")`: організація (умолчання користувача → договір → фолбек
  перша), валюта документа, курси документа і взаєморозрахунків (`КурсыВалют` на дату),
  вид угоди, контактна особа, підрозділ, банківські рахунки (`СтруктурнаяЕдиница` = осн.
  банк. рахунок — **не склад**! склад документа — `СкладГруппа`), `УчитыватьНДС` за
  платником ПДВ, `СуммаВключаетНДС` за типом цін, `АвторасчетНДС`. `ВалютаДокумента` —
  **валюта пристрою** (не регламентована), договір — валютний (див. «Валюта» нижче).
- **Пристрій перекриває**: `ТипЦен`, `СкладГруппа`, `Ответственный`, `ДоговорКонтрагента`
  (ставляться до/після шапки так, щоб умолчання їх не затерли).
- **Рядки** (`ЗаполнитьТовары`) — одиниця `ЕдиницаИзмерения = Номенклатура.
  ЕдиницаХраненияОстатков` (саме довідник `ЕдиницыИзмерения` — тип поля ТЧ;
  `БазоваяЕдиницаИзмерения` — класифікатор, присвоєння дає порожнє поле!),
  `Коэффициент` — з одиниці; далі `ОбработкаТабличныхЧастей.ЗаполнитьСтавкуНДСТабЧасти
  (…, "Реализация")` + `ЗаполнитьЕдиницуМестТабЧасти` + `РассчитатьСуммуНДСТабЧасти`,
  `Размещение = СкладГруппа`. Каталог (`GET /products`) віддає назву одиниці теж з
  `ЕдиницаХраненияОстатков` — щоб додаток показував ту саму одиницю, що й документ.
  **Ціна рядка — snapshot з пристрою**, за типом цін не перезаповнюється (контракт Vendo).
- **Зміна контрагента** при `PUT` — `ЗаполнениеДокументов.ПриИзмененииЗначенияКонтрагента`
  (перепідбір договору/організації), далі примусово ставиться **валютний** договір і валюта
  документа + курс взаєморозрахунків (валюти пристрою).

## Валюта (#35)

Пристрій **одновалютний** — валюта задається в картці (`венд_МобильныеУстройства.Валюта`,
узагальнений `СправочникСсылка`, обмежений на формі до `Валюты`; при створенні
автозаповнюється регламентованою). Весь каталог, кошик і замовлення пристрою — у ній.
Бекенд усюди віддає **числовий код ISO 4217 рядком** (`"980"`=грн, `"840"`=USD…), фронт
малює символ; форматування — лише на фронтенді (жодних `"N ₴"` з бекенду).

- **`GET /products`** — ціни `ЦеныНоменклатуры` (мають власний вимір `Валюта`, можуть бути
  в різних валютах) перераховуються у валюту пристрою: `МодульВалютногоУчета.
  ПересчитатьИзВалютыВВалюту(Цена, ВалютаЦены, ВалютаПристрою, КурсИсх, КурсЦел, КратнІсх,
  КратнЦел)`, курси — `ПолучитьКурсВалюты(валюта, ТекущаяДатаСеанса())`. Курс цільової
  валюти рахується раз, курси джерел кешуються в `Соответствие` (немає звернень до БД у
  циклі). Відповідь: `currency` = `Устройство.Валюта.Код` (одна на всі позиції).
- **Замовлення** — `ВалютаДокумента` = валюта пристрою; курс/кратність взаєморозрахунків
  за нею. `СериализоватьЗаказ`: `total` — **число** (не рядок), `currency` =
  `ВалютаДокумента.Код` (заморожена; старі документи без валюти → `"980"`).
- **Договір за валютою** — `ПодобратьДоговорПоВалюте(Контрагент, Валюта)` шукає договір
  «з покупцем» із `ВалютаВзаиморасчетов = Устройство.Валюта`; пріоритет —
  `ОсновнойДоговорКонтрагента`, якщо він теж підходить, інакше будь-який чинний. Немає
  такого → `СохранитьЗаказ`
  повертає `Неопределено` з текстом → стандартний **400** («немає договору у валюті …»),
  замовлення лишається в черзі застосунку. Тобто валюта пристрою мусить збігатися з валютою
  договорів його контрагентів.
- **`GET /customers`** — `debt` лишається в **управлінській** валюті (зведений
  `СуммаУпрОстаток`); `debtCurrency` = `Константы.ВалютаУправленческогоУчета.Получить().Код`
  (не перераховується у валюту прайсу).

## Потребує вивірки в цільовій УТ

- Умолчання користувача публікації (`ПользовательСеанса`): основна організація/склад/
  підрозділ — якщо задані користувачу `vendo` в УТ, підхопляться автоматично.
- **Валюта (#35):** `ЗаполнитьШапкуДокумента` з **не-регламентованою** валютою документа
  (чи коректно проставляє курс документа/взаєморозрахунків); заповненість `КурсыВалют` на
  дати замовлень (немає курсу → `ПолучитьКурсВалюты` бере останній зріз); поведінка УТ при
  проведенні, коли валюта документа = валюті договору ≠ регламентованій.

**Проведення** документа — **реалізовано** (`ЗаписатьЗаказ`→`НужноПровести`): за прапорцем
пристрою `АвтоПроведение` (або явним статусом «Проведено» в тілі) запис іде
`Записать(РежимЗаписиДокумента.Проведение)` з рухами, інакше `Записать(Запись)`. Запис і
проведення — в одній транзакції. Потребує перевірки в цільовій УТ: контроль залишків чи
незаповнені обов'язкові поля можуть блокувати проведення (тоді `POST`/`PUT` → 400 з описом).
