# Публікація HTTP-сервісу Vendo на IIS

Перевірений мінімум налаштувань веб-шару для бази УТ із міні-конфігурацією Vendo.
Сам сервіс (контракт, помилки, права) описано в [DEVELOPER.md](../DEVELOPER.md).

Для **Apache** цей каталог не потрібен: `httpErrors` — суто IIS-механізм, решту
Конфігуратор налаштовує сам.

## Що публікуємо

| | |
|---|---|
| HTTP-сервіс | `венд_МобильноеПриложение` |
| rootURL | `vendo` |
| Методів | 15 (у 12 шаблонах URL) |
| Адреса ендпоінта | `https://<host>/<база>/hs/vendo/<шлях>` |
| Перевірка живості | `GET …/hs/vendo/health` → `{"status","time","version","minAppVersion"}` |

## Передумови

1. **Компоненти IIS**: Web Server → Application Development → **ISAPI Extensions** + **ISAPI Filters**.
2. **ISAPI and CGI Restrictions** (вузол сервера в IIS Manager): `wsisapi.dll` — **Allowed**.
3. **Бітність пулу застосунків збігається з `wsisapi.dll`**:
   - `C:\Program Files (x86)\…\wsisapi.dll` (32 біт) → `Enable 32-Bit Applications = True`
   - `C:\Program Files\…\wsisapi.dll` (64 біт) → `False`
4. Identity пулу читає `bin` платформи; для **файлової** бази — ще й пише в каталог бази.

## Крок 1. Публікація з Конфігуратора

Адміністрування → Публікація на веб-сервері. Конфігуратор створює каталог публікації
(типово `C:\inetpub\wwwroot\<база>\`) із двома файлами:

- `default.vrd` — що публікуємо і під яким користувачем ІБ;
- `web.config` — прив'язка IIS до `wsisapi.dll`.

Обидва **генеровані**. Руками правлять лише точково — далі.

## Крок 2. `web.config` — обов'язковий `httpErrors`

Взірець: [`web.config`](web.config).

```xml
<httpErrors existingResponse="PassThrough" />
```

Без цього рядка IIS **підміняє тіло** будь-якої відповіді 4xx/5xx власною HTML-сторінкою.
Статус-код зберігається, а JSON зникає. Помилки сервіс віддає JSON-тілом (типово
`{success: false, message}`) — без `PassThrough` додаток замість тексту помилки дістає
`Failed to execute 'json' on 'Response'`.
Гейт версії ([#66](https://github.com/Lion-killer/Vendo/issues/66)) віддає `426` з тілом
`{error: "app_too_old", minAppVersion, appVersion}`, і без
`PassThrough` користувач не побачить, чому додатку відмовили.

> ⚠️ **Тільки ASCII.** Файл оголошений `encoding="UTF-8"`. Кирилиця в коментарі, збережена
> Блокнотом в ANSI/1251, робить його невалідним XML — і IIS кладе **всю** публікацію
> (`500.19 · 0x8007000d`), включно з веб-клієнтом. Те саме дає `--` усередині коментаря.
> Це реальний інцидент, а не гіпотеза: коментарі в `web.config` пиши латиницею.

IIS перечитує `web.config` сам — `iisreset` не потрібен. Після правки перевір:

```powershell
$p = 'C:\inetpub\wwwroot\<база>\web.config'
[xml](Get-Content $p -Raw) | Out-Null; 'XML OK'
$dll = ([regex]'scriptProcessor="([^"]+)"').Match((Get-Content $p -Raw)).Groups[1].Value
$dll; Test-Path $dll
```

## Крок 3. `default.vrd` — що публікуємо і під ким

Взірець: [`default.vrd`](default.vrd) — публікує **лише** HTTP-сервіс `vendo`, усе інше
вимкнено. Перед копіюванням на сервер заміни шлях до бази в `ib=` (для серверної:
`Srvr=…;Ref=…;`) і пароль `CHANGEME`.

Семантика — за керівництвом адміністратора платформи (розділ про `default.vrd`;
[дзеркало 1Ci KB](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/)):

| Елемент / атрибут | У взірці | Що робить |
|---|---|---|
| `point enable` | `false` | тонкий/мобільний/веб-клієнт (`/e1cib/*`) не публікуються — на корені бази браузер отримує помилку, а не вікно входу; на HTTP-сервіси не впливає (у них власні прапорці) |
| `ib` | `File=…;Usr=vendo;Pwd=…` | рядок з'єднання; креденшели користувача публікації **відкритим текстом** |
| `ws enable` + `pointEnableCommon` | `false` | SOAP-сервіси вимкнені повністю — обидва атрибути за замовчуванням `true`! |
| `httpServices publishByDefault` | `false` | білий список: доступні лише явно перелічені `<service enable="true">`. Без атрибута публікуються **всі** HTTP-сервіси конфігурації |
| `publishExtensionsByDefault` | `false` | сервіси з приєднаних розширень не публікуються (це і дефолт — атрибут для явності) |
| `service reuseSessions` | `autouse` | пул сеансів ІБ — див. нижче |
| `standardOdata enable` | `false` | стандартний OData REST (`/odata/standard.odata`) вимкнено |
| `analytics enable` | `false` | 1С:Аналітика вимкнена. ⚠️ **Відсутність елемента означає «увімкнено»** — рядок не зайвий |

Чого у взірці **нема** — і це правильно, бо для цих елементів відсутність = «вимкнено»:
`<debug>` (налагодження через публікацію — ніколи не вмикати назовні), `<openid>`,
`<openidconnect>`, `<progressiveWebApplication>`, `pubdst*` (роздача дистрибутивів клієнта).

### `reuseSessions="autouse"` — чому не `dontuse`

З `dontuse` платформа на **кожен** запит створює сеанс ІБ (повна ініціалізація,
`УстановкаПараметровСеанса()`) і завершує його після відповіді. Офіційний вимір 1С
([блог розробників](https://wonderland.v8.1c.ru/blog/povyshenie-proizvoditelnosti-veb-servisov/)):
100 послідовних викликів — 29.9 с без пулу проти ~3 с із пулом, тобто **~10×**.
Сервіс Vendo — stateless (токен пристрою перевіряється в кожному запиті, стану сеансу
нема), тож пул безпечний. `sessionMaxAge="20"` (секунди простою до завершення сеансу)
достатньо: додаток пінгує `/health` типово кожні 15 с (інтервал керується налаштуваннями
бекенду, #68) — пул лишається теплим. `poolSize`/
`poolTimeout` — згенеровані Конфігуратором значення, шкоди не роблять; при вичерпанні
пулу запит чекає `poolTimeout` секунд і відбивається 406.

Зміна цих атрибутів набирає сили після перезапуску робочого процесу — для файлової
бази на IIS це **Recycle пулу застосунків**.

### Креденшели й супровід

- Пароль у `ib=` — відкритим текстом, тому реальні `.vrd` **не комітяться**
  (`.gitignore`: `*.vrd`; єдиний виняток — цей взірець із `CHANGEME`).
- Користувач публікації окремий, лише з роллю `венд_HTTPСервис`
  (див. [DEVELOPER.md](../DEVELOPER.md#права-роль-і-користувач-публікації)); інтерактивний
  вхід йому недоступний — викрадені креденшели не дають зайти в базу клієнтом.
- ⚠️ **Перепублікація з Конфігуратора регенерує `default.vrd`** (і може перезаписати
  `web.config`) — ручні правки (reuseSessions, вимкнені сервіси, `httpErrors`) мовчки
  зникають. Захарденені файли далі супроводжують руками; взірці в цьому каталозі — страховка.
- `.vrd` лишається нечитабельним ззовні лише поки прив'язка обробника IIS покриває
  `path="*"` — якщо її зняти, IIS може віддати файл як статичний, разом із паролем.
  Тримай NTFS-права каталогу публікації мінімальними.

## Крок 4. Перевірка

```bash
B=https://<host>/<база>
curl -sS -o /dev/null -w "%{http_code}  %{time_total}s\n" "$B/"                        # веб-клієнт (із взірцевим .vrd вимкнено — очікуй помилку, не 200)
curl -sS -o /dev/null -w "%{http_code}  %{time_total}s\n" "$B/hs/vendo/health"         # сервіс
curl -sS -o /dev/null -w "%{http_code}  %{time_total}s\n" "$B/hs/nemaye-takogo/health" # контроль
curl -sS "$B/hs/vendo/health"                                                          # тіло JSON
```

## Тріаж-евристика

Третій запит навмисне б'є в **неіснуючий** HTTP-сервіс. Він і є діагностом:

| Контрольний запит | Висновок |
|---|---|
| **404** | IIS і 1С живі, маршрутизація працює → причина в самому сервісі, базі або правах |
| **500** | до 1С не дійшло — падає IIS-модуль → `web.config`, ISAPI Restrictions, бітність пулу |

Час відповіді додає інформації: жива 1С відповідає сотнями мілісекунд і повільніше
(перше з'єднання з базою — секунди). Миттєвий 500 за ~0.1 с означає, що з'єднання з
базою навіть не пробувалось.

Щоб побачити справжній текст помилки: IIS Manager → застосунок → **Error Pages → Edit
Feature Settings → Detailed errors**. Тимчасово: цей режим віддає назовні внутрішні
шляхи. Після діагностики поверни `DetailedLocalOnly`.

## Типові помилки

| Симптом | Причина | Що робити |
|---|---|---|
| `500.19`, код `0x8007000d`, «Файл конфигурации создан в неправильном формате XML» | `web.config` не парситься: не-ASCII байти або `--` у коментарі | лишити тільки ASCII, зберегти в UTF-8 |
| `500.19`, код `0x80070021`, «This configuration section cannot be used at this path» | секцію заблоковано на рівні сервера | `%windir%\system32\inetsrv\appcmd unlock config -section:system.webServer/handlers` |
| Усе під `/<база>/` віддає 500, зокрема неіснуючий сервіс і веб-клієнт | запит не доходить до 1С | тріаж-евристика вище |
| HTML-сторінка замість JSON на 401/404/426/500 | нема `httpErrors existingResponse="PassThrough"` | додати рядок у `<system.webServer>` |
| `404` на `/hs/vendo/…` | у `default.vrd` сервіс без `enable="true"`, rootURL ≠ `vendo`, або файл збережено не в UTF-8 (кириличне ім'я сервісу не збігається з метаданими) | виправити `.vrd` за взірцем, Recycle пулу |
| після перепублікації з Конфігуратора все «злетіло» | публікація регенерує `default.vrd` (і може перезаписати `web.config`) — ручні правки зникли | відновити з взірців цього каталогу |
| `406 Not Acceptable` під навантаженням | вичерпано пул сеансів (`poolSize`), запит не дочекався за `poolTimeout` | збільшити `poolSize`; для файлової бази — це очікувана межа паралельності |
| `401` на всі методи, включно з `/health` | креденшели в `.vrd` або роль користувача публікації | звірити користувача й роль `венд_HTTPСервис` |
| `405` на `PUT`/`DELETE /orders/{id}` | запит перехоплює `WebDAVModule` | зняти модуль — див. нижче |
| Повільний 500 зі сторінкою помилки **1С** (не IIS) | база недоступна: монопольний режим, оновлення конфігурації БД, не запущений агент сервера | закрити Конфігуратор / стартувати службу |

### WebDAV перехоплює PUT і DELETE

Додавати **лише якщо** `PUT`/`DELETE` реально віддають `405` — зайві рядки в файлі, який
здатен покласти всю публікацію, того не варті:

```xml
<handlers>
    <remove name="WebDAV" />
    <!-- … 1C Web-service Extension … -->
</handlers>
<modules>
    <remove name="WebDAVModule" />
</modules>
```
