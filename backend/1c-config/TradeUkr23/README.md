# Міні-конфігурація 1С — бекенд Vendo

Окрема **конфігурація-постачальник** для **1С «Управління торгівлею для України»**,
яка виступає реальним бекендом мобільного застосунку Vendo замість демонстраційного
Node-сервера (`../../mock/`). Об'єкти оформлено окремою міні-конфігурацією, яку
**вбудовують у цільову УТ через «Порівняння/об'єднання конфігурацій»**.

- Префікс об'єктів: `венд_`
- Цільова база: УТ для України 2.3
- Сумісність: `Version8_3_19` — працює навіть у режимі сумісності **8.2.13**
  (HTTP-сервіс вимагає лише платформи **≥ 8.3.5**)

> 🛠️ Технічна частина для розробника 1С (типи реквізитів, запити, локалізація,
> BSL-нюанси) — у [DEVELOPER.md](DEVELOPER.md).

## Що це дає

Після об'єднання з УТ застосунок Vendo отримує дані з реальної бази:

- номенклатуру з цінами (за типом цін пристрою), залишками й фото;
- дерево груп номенклатури (категорії);
- контрагентів з боргом, адресою, телефоном, контактними особами;
- замовлення менеджера за період;
- приймає й створює документи «Замовлення клієнта» (`ЗаказПокупателя`).

Дані фільтруються **за пристроєм** (планшетом): кожен застосунок бачить лише свій
склад, тип цін, менеджера, своїх контрагентів і свою номенклатуру.

## Як вбудувати в цільову УТ

1. Завантажити конфігурацію з XML-файлів (`LoadConfigFromFiles`) у **пусту базу**.
2. Зберегти конфігурацію у **CF**-файл (Конфігуратор → «Зберегти конфігурацію у файл»).
3. У базі цільової УТ **об'єднати** цей CF із конфігурацією (**Порівняння й об'єднання
   конфігурацій** → перенести `венд_*` об'єкти).
4. Оновити конфігурацію БД, опублікувати HTTP-сервіс на веб-сервері (Apache/IIS) —
   отримати URL виду `https://<сервер>/<база>/hs/vendo`.
5. Завести записи довідника «Мобільні пристрої» (див. нижче).

> Об'єднання працює навіть якщо УТ у режимі сумісності 8.2.13 — обмежень немає.
> Чому код запрацює лише **після** об'єднання — у [DEVELOPER.md](DEVELOPER.md).

## Довідник «Мобільні пристрої» (`венд_МобильныеУстройства`)

Центральна сутність налаштування: **один запис = один планшет/користувач застосунку**.
Заповнюється оператором УТ:

| Реквізит | Призначення |
|---|---|
| `ИдентификаторУстройства` | GUID із QR-коду для автентифікації пристрою |
| `Склад` | склад — джерело залишків |
| `ТипЦен` | тип цін, за яким застосунок бачить ціни |
| `Менеджер` | відповідальний користувач (`Пользователи`) |
| ТЧ `Контрагенты` | доступні клієнти — окремий контрагент **або ціла група** |
| ТЧ `Товары` | доступна номенклатура — окремий товар **або ціла група** |

**Вибір групи чи елемента.** У табличних частинах «Контрагенти» / «Товари» можна
вказати або конкретний елемент, або **групу** (папку):

- група → усі товари/контрагенти всередині неї, включно з підгрупами на будь-якій глибині;
- елемент → лише він.

Перетини не дублюються: кожен товар/контрагент потрапляє в застосунок **один раз**,
навіть якщо вказати і групу, і її підгрупу, і окремий елемент.

> У формі елемента кожне поле обмежене правильним довідником: Склад обирається лише
> зі `Склады`, Менеджер — лише з `Пользователи` тощо.

## HTTP-сервіс — що віддає

`венд_МобильноеПриложение` (rootURL `vendo`), контракт **1:1** з демо-бекендом.
Пристрій передається заголовком `X-Device-Id` або параметром `?device=<GUID>`.

| Метод | Шлях | Опис |
|---|---|---|
| GET/HEAD | `/health` | перевірка доступності (без авторизації) |
| POST | `/auth` | `{deviceId, pairingCode}` → перевірка пристрою + bearer-токен |
| GET | `/products` | номенклатура з ціною, залишком (склад − резерв), штрихкодом, фото |
| GET | `/products/{id}/image` | бінарне фото товару (`?v=` — інвалідація кешу) |
| GET | `/categories` | повне дерево груп номенклатури |
| GET | `/customers` | контрагенти з боргом, адресою, телефоном, контактними особами |
| GET | `/orders?startDate=&endDate=` | замовлення менеджера за період |
| POST | `/orders` | upsert `ЗаказПокупателя` за клієнтським GUID (ідемпотентно) |
| PUT | `/orders/{id}` | оновити замовлення (за GUID) |
| DELETE | `/orders/{id}` | помітка на видалення (за GUID) |

Повідомлення сервісу віддаються 3 мовами (uk / ru / en) за заголовком `Accept-Language`.

## Що потребує вивірки в цільовій базі

Створення `ЗаказПокупателя` спирається на реквізити, які варто звірити з обліковою
політикою бази (`Организация`, договір, проведення документа). Конкретні місця в коді
й поточна поведінка — у [DEVELOPER.md](DEVELOPER.md#потребує-вивірки-в-цільовій-ут).
