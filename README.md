# Vendo

Мобільний застосунок для торгових представників (B2B) — каталог товарів, контрагенти та оформлення замовлень «у полі». Прототип з українським інтерфейсом, **offline-first** логікою та збіркою під Android через Capacitor.

> ⚠️ Прототип/демо: автентифікація імітована (mock-токен), дані сідяться з фікстури. Не для продакшну «як є».

## Стек

| Частина | Технології |
|---|---|
| **Backend** | Node.js + Express 5, SQLite (`better-sqlite3`) |
| **Frontend** | React 17, Vite 8, inline-стилі (без CSS-фреймворку) |
| **Mobile** | Capacitor 8 (Android) |

## Структура

```
Vendo/
├─ backend/            # REST API над SQLite
│  ├─ server.js        # точка входу (порт 3000)
│  ├─ db.js            # схема SQLite + одноразовий сід із db.json
│  ├─ routes/api.js    # ендпоінти (prepared statements)
│  └─ data/db.json     # сід-фікстура (товари, клієнти, категорії, замовлення)
├─ frontend/           # React + Vite (+ Android-проект Capacitor)
│  └─ src/
│     ├─ App.jsx       # оркестратор: навігація, стан, offline-логіка
│     ├─ api/          # REST-клієнт + локальне сховище чернеток
│     ├─ screens/      # екрани (login, dashboard, catalog, customers, order...)
│     ├─ components/   # Icon, BottomNav, Snackbar, Badge
│     └─ theme.js      # токени світлої/темної теми
├─ design/             # початковий single-file мокап
└─ start.bat           # запуск backend + frontend одночасно (Windows)
```

## Запуск

Потрібен Node.js 18+ (розроблялось на Node 24).

### Швидкий старт (Windows)

```bat
start.bat
```
Підніме backend (http://localhost:3000) і frontend (http://localhost:5173) у двох вікнах.

### Вручну

**Backend:**
```bash
cd backend
npm install
npm start
```
При першому запуску `db.js` створює `data/vendo.db` і сідить його з `data/db.json`.

**Frontend:**
```bash
cd frontend
npm install
npm run dev -- --host
```
`--host` відкриває доступ у локальній мережі (для тесту на пристрої/емуляторі).

### Збірка та Android

```bash
cd frontend
npm run build          # → dist/
npx cap sync android   # синхронізувати веб-збірку в Android-проект
```

> Для реального пристрою/емулятора зверніть увагу: у [`src/api/client.js`](frontend/src/api/client.js) адреса API захардкоджена на `http://localhost:3000`. Для Android-емулятора — `10.0.2.2`, для реального пристрою — IP комп'ютера в локальній мережі.

## API

Базовий шлях: `/api`

| Метод | Шлях | Опис |
|---|---|---|
| `POST` | `/auth` | Імітація логіну (повертає mock-користувача й токен) |
| `GET` | `/products` | Список товарів |
| `GET` | `/categories` | Категорії |
| `GET` | `/customers` | Контрагенти |
| `GET` | `/orders` | Замовлення (фільтр `?startDate=&endDate=`, формат `YYYY-MM-DD`) |
| `POST` | `/orders` | Створити замовлення |
| `PUT` | `/orders/:num` | Оновити замовлення |
| `DELETE` | `/orders/:num` | Видалити замовлення |

## Особливості архітектури

- **Нормалізоване зберігання замовлень.** У БД замовлення містять лише посилання (`customerId`, `productId`), а повний об'єкт з товарами/клієнтом збирається «на льоту» при віддачі (гідратація). Жодного дублювання даних.
- **Фіксація ціни (snapshot).** Ціна кожної позиції зберігається на момент оформлення замовлення — подальша зміна прайсу в каталозі не впливає на історичні суми.
- **Транзакційні записи.** Створення/оновлення замовлення відбувається в транзакції SQLite (атомарно, без втрати даних при конкурентних запитах).
- **Стійка нумерація.** Номери замовлень (`ЗМ-NNNN`) видаються з персистентного лічильника й не повторюються після видалень.
- **Offline-first.** Дані кешуються в `localStorage`; фоновий пінг бекенду кожні 15 с перемикає онлайн/офлайн. Чернетки й невідправлені замовлення зберігаються локально й синхронізуються при відновленні зв'язку.

## Скидання бази

Зупиніть backend і видаліть файли БД — вони пересіються з `db.json` при наступному запуску:

```bash
rm backend/data/vendo.db backend/data/vendo.db-wal backend/data/vendo.db-shm
```
